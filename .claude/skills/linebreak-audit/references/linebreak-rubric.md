# 페이지 줄바꿈 품질 루브릭 (linebreak-audit)

> 이 루브릭은 "HTML 페이지(특히 발표 자료)의 줄바꿈이 **얼마나 어색한가**"를 렌더 측정값에
> 근거해 페이지(또는 덱) 단위로 채점하고, 개선 후 **실제로 나아졌는가**를 판정하기 위한
> 기준이다. 한국어 조판 특성(어절 단위 줄바꿈, `word-break: keep-all`)을 1급 기준으로 둔다.
>
> 핵심 관점:
> - 줄바꿈은 **렌더 결과**다. 컨테이너 너비·폰트·언어 규칙이 합쳐져 정해진다. 그래서
>   `scripts/scan_linebreaks.js` 가 실제 줄 박스를 측정한 **사실(scan.json)** 위에서 채점한다.
> - 밸런싱·keep-all을 적용하면 줄바꿈이 눈에 띄게 **바뀐다**. 이는 실패가 아니라 **성공 신호**다.
> - 모든 텍스트가 대상은 아니다. **제목·짧은 블록**(헤딩/부제/불릿/캡션/큰 숫자)이 1차 대상이고,
>   긴 본문 문단은 고아·넘침만 본다(밸런싱 대상이 아니므로).

---

## 0. 입력

평가자는 다음을 근거로 판정한다(스캔 JSON은 필수, 스크린샷으로 교차 확인):

- `scan.json` — `scan_linebreaks.js` 출력. 각 문제 요소의 줄 수·줄별 너비·렌더된 줄 텍스트·
  문제 유형·제공정 severity·중요도(weight)·페이지 점수가 들어 있다. **[필수, 1급 근거]**
- 페이지/슬라이드 **스크린샷** — 스캔이 못 보는 맥락(의도된 줄바꿈, 디자인 의도)을 사람 눈으로
  교차 확인하고 **오탐을 걸러내기 위해** 본다. **[필수]**
- (개선 검증 시) `before.scan.json` ↔ `after.scan.json` 두 벌과 두 스크린샷.

스캐너는 **렌더 측정값**을 주고, 스크린샷은 **의도/맥락**을 준다. 둘이 어긋나면 스크린샷의
관찰을 우선해 스캐너 항목을 오탐으로 기각할 수 있다(§5).

---

## 1. 문제 분류 (무엇을 "어색한 줄바꿈"으로 보는가)

| ID | 유형(scan type) | 정의 | 스캐너 탐지 방식 | 기본 severity |
|----|----------------|------|------------------|:---:|
| **P1** | `midword_break` | **어절(단어) 중간이 끊김.** 한국어에서 가장 거슬리는 문제 — `keep-all` 누락의 전형(예: `방\|법`, `이\|해`, `확대되\|고`). | 줄 경계의 두 글자가 모두 공백이 아니고 한글/라틴 단어 글자면 끊김으로 판정. | 짧은 블록 **3** / 긴 문단 2 |
| **P2** | `orphan` | **마지막 줄 외톨이.** 마지막 줄이 한 단어·짧은 토막으로 외롭게 떨어짐. | `마지막 줄너비/가장 넓은 줄 < 0.45` 또는 마지막 줄 단어 1개. | 1–3(짧고 외로울수록↑) |
| **P3** | `imbalance` | **줄 길이 불균형.** 줄들이 들쭉날쭉(가장 긴 줄 대비 편차 큼). 고아로 이미 잡힌 건 제외. | `(max−min)/maxW > 0.5`. | 1–2 |
| **P4** | `overflow` | **넘침/잘림.** 컨테이너를 넘치거나 `…`/고정 높이로 잘림. 레이아웃 파손. | `scrollWidth>clientWidth` 또는 `scrollHeight>clientHeight`(overflow≠visible). | **3** |
| **P5** | `semantic_split` | **의미 단위 분리.** 숫자와 그 단위가 줄을 넘어 갈림(예: `4` / `조`). | 공백 경계지만 앞 토큰이 숫자로 끝나고 뒤가 단위로 시작. | 2 |
| **P6** | `excess_lines` | **과다 줄 수.** 짧은 제목이 여백이 많은데도 4줄 이상으로 쪼개짐. | `line_count≥4 & 어절≤10 & 평균 채움<55%`. | 1 |

> 의도적으로 줄을 끊는 **연출**(시 형식, 강조용 `<br>`, 디자인된 줄나눔)은 문제가 아니다(§5).
> P1·P4는 거의 항상 진짜 문제이고, P3·P6은 맥락을 더 본다.

---

## 2. severity와 중요도(weight)

**severity (한 문제의 거슬림 정도):** `3` 심각(즉시 거슬림) · `2` 눈에 띔 · `1` 경미.

**weight (요소의 시각적 중요도):** 같은 문제라도 큰 제목에서 더 치명적이다.

| 역할(role) | 판정 | weight |
|---|---|:---:|
| hero | `h1` 또는 폰트 ≥ root의 2.2배 | **3** |
| title / subtitle | `h2`,`h3`,`blockquote` 또는 폰트 ≥ root의 1.4배 | **2** |
| body | 그 외(본문 `p`·`li`·캡션·라벨) | **1** |

---

## 3. 채점 (페이지 점수 0–100)

**블록 청결도** `cleanliness = max(0, 1 − min(3, Σseverity) / 3)`
→ severity-3 문제 하나면 0, 무결하면 1.

**페이지 점수** `page_score = round( 100 × Σ(weight × cleanliness) / Σ(weight) )`
- 합산 대상은 **줄바꿈이 일어난 블록**(2줄 이상 또는 넘침). 한 줄짜리 정상 텍스트는 분모에서 제외.
- 줄바꿈 블록이 하나도 없으면 `page_score = 100`(고칠 게 없음).

`scan.json`이 이 점수를 이미 계산해 둔다(`summary.page_score`). 평가자는 스크린샷으로 오탐을
걸러낸 뒤(§5) 남은 문제로 **확정 점수**를 보고한다(오탐을 떨궜으면 그만큼 올라간다).

| 점수 | 등급 | verdict | 의미 |
|---|:---:|---|---|
| 90–100 | A | `clean` | 사실상 깔끔. 손댈 것 거의 없음. |
| 80–89 | B | `minor` | 사소한 잔여. 큰 제목만 손봐도 충분. |
| 60–79 | C/D | `needs_work` | 눈에 띄는 문제 다수. 개선 권장. |
| <60 | F | `poor` | 줄바꿈이 전반적으로 어색. 개선 필요. |

---

## 4. 하드 게이트 (개선 후 점검 — 위반 시 점수와 무관하게 `regression`)

수정이 **해를 끼치지 않았는지**부터 본다. 점수가 아니라 통과/실패다.

| ID | 게이트 | 위반 = 실패 |
|----|--------|-------------|
| **G1 · 내용 보존** | 글자·단어·순서가 누락/중복/변형되지 않음 | 후에서 텍스트가 사라지거나 깨짐. |
| **G2 · 넘침 무발생** | 수정이 새로운 넘침/잘림을 만들지 않음 | keep-all/nowrap 등으로 후에 새 overflow 발생. |
| **G3 · 줄 수 비증가** | 같은 블록의 줄 수가 늘지 않음(밸런싱은 높이 불변이 원칙) | 후 줄 수 > 전 줄 수(보통 설정 오류). |
| **G4 · 부작용 없음** | 정렬·간격·인접 요소·반응형이 망가지지 않음 | 후 스크린샷에서 레이아웃 붕괴. |

> 긴 본문에 `text-wrap: balance`를 잘못 적용하면 G3을 위반하기 쉽다 — 본문은 대상이 아니다.

## 5. 오탐 방지 / 특수 케이스

스캐너 항목을 그대로 믿지 말고 스크린샷으로 다음을 먼저 분기한다.

- **의도된 줄바꿈(연출):** 강조·리듬을 위한 수동 `<br>`, 시·가사·구호 형식. → 문제 아님. 기각.
- **한 줄 텍스트 / 이미 균형:** 분모에서 빠지거나 cleanliness 1. "변화 없음"을 실패로 적지 말 것.
- **밸런싱 후 shrink-wrap:** 밸런싱하면 요소가 가장 긴 줄로 줄어든다. 줄들이 고르면 정상(높은 점수).
- **프레임워크 덱의 숨은 슬라이드:** reveal.js 등은 활성 슬라이드만 렌더한다. 숨은 섹션은 측정
  불가(0 크기)라 스캔에서 빠진다 → `__lbRevealAll()`로 펼친 뒤 스캔했는지 확인(rendering.md).
- **긴 본문 문단:** 어절 끊김·불균형은 본문에선 흔하고 덜 거슬린다. 스캐너도 긴 블록은
  고아·넘침만 남긴다. 본문을 과교정하지 말 것.
- **숫자/단위가 안 갈린 경우:** `2025 년`이 한 줄에 있으면 P5 아님(실제로 갈렸을 때만).
- **CJK(중국어/일본어) 본문:** 글자 사이 줄바꿈이 정상 조판인 언어다. 스캐너는 한글이 낀
  경계만 어절 끊김으로 본다 — 순수 한자/가나 사이는 건드리지 않는다.

## 6. 개선 검증 (전/후 비교)

수정 후 같은 조건으로 다시 스캔(`after.scan.json`)해 비교한다.

**개선 성공(`improved`) 판정:** 다음을 모두 만족.
1. 하드 게이트(§4) 위반 없음.
2. `after.page_score ≥ 85` **그리고** severity-3 문제 0개.
3. `after.page_score > before.page_score` (점수가 실제로 올랐다).

- 위 1–3 충족 → `verdict: "improved"`. 적용 유지.
- 점수는 올랐으나 85 미만이거나 sev-3 잔존 → `verdict: "marginal"`. §7 조치로 한 번 더 반복.
- 게이트 위반 또는 점수 하락 → `verdict: "regression"`. 수정을 되돌리고 원인 분석.
- 전이 이미 ≥90이었다 → `verdict: "noop"`(고칠 게 없었음). 정상.

보고에는 **Δ(점수 변화)**, 유형별 문제 수 변화(예: `midword 7→0`), 남은 항목을 적는다.

## 7. 점수가 낮을 때 표준 조치 (fixing.md 요약)

| 문제 | 조치 |
|------|------|
| P1 어절 끊김 | 대상(또는 덱 전역 텍스트)에 `word-break: keep-all; overflow-wrap: anywhere;` (`break-all` 제거) |
| P2 고아 / P3 불균형 | 짧은 블록에 wrap-balancer 적용(`data-br-balance`) — 균형·외톨이 해소 |
| P4 넘침/잘림 | `white-space: normal`로 줄바꿈 허용 / 고정 `height`·`width` 제거 / `overflow-wrap: anywhere` |
| P5 의미 분리 | 숫자와 단위 사이를 `&nbsp;`로 묶거나 `<wbr>`로 끊을 지점 지정 |
| P6 과다 줄 수 | 컨테이너 폭↑ 또는 폰트↓ 후 wrap-balancer로 균형 |

가장 비용 대비 효과가 큰 것은 **덱 전역 keep-all 한 번**(P1 대량 해소) + **짧은 블록 일괄
밸런싱**(P2/P3 해소)이다. P4는 요소별 레이아웃 수정이라 개별 대응한다.

## 8. 출력 계약 (평가 에이전트가 반환할 JSON)

```json
{
  "verdict": "clean | minor | needs_work | poor | improved | marginal | regression | noop",
  "page_score": 0,
  "grade": "A|B|C|D|F",
  "gates": { "G1_content": true, "G2_no_overflow": true, "G3_line_count": true, "G4_no_side_fx": true },
  "totals": { "scanned": 0, "wrapped": 0, "findings": 0, "by_problem": {"midword_break": 0, "orphan": 0} },
  "top_findings": [
    { "where": "section#s1[0] h1.hero-title", "role": "hero", "problems": ["midword_break(3)","orphan(3)"],
      "lines": ["...","..."], "fix": "keep-all + wrap-balancer", "confirmed": true }
  ],
  "false_positives": [
    { "where": "...", "reason": "강조용 의도된 줄바꿈이라 문제 아님" }
  ],
  "delta": { "before_score": 0, "after_score": 0, "resolved": {"midword_break": 7}, "remaining": [] },
  "suggestions": ["가장 효과 큰 다음 조치"],
  "reasoning": "1–3문장 한국어 요약"
}
```

규칙:
- `page_score`는 0–100 정수. 단일 감사면 `delta`는 생략 가능, 개선 검증이면 `delta` 필수.
- `gates` 중 하나라도 `false`면 `verdict`는 `regression`.
- 스크린샷에서 직접 관찰되는 사실만 적는다. 스캐너 수치(`scan.json`)가 있으면 인용한다.
- `top_findings`/`suggestions`/`reasoning`은 **한국어**. 근거 없는 추측 금지.
